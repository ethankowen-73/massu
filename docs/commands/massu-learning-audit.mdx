# /massu-learning-audit

Validate that the CR-34 auto-learning protocol is working effectively by checking memory coverage for canonical rules, pattern scanner coverage for documented incidents, failure recurrence rates, and session quality statistics.

## Usage

```
/massu-learning-audit
```

## Audit Sections

### Section 1: Memory Coverage Check
For each canonical rule (CR-1 through CR-12 and beyond), query memory to verify entries exist. Every critical CR should have at least one memory entry.

### Section 2: Pattern Scanner Coverage
For each documented incident, verify the pattern scanner catches the bad pattern. Checks that `scripts/pattern-scanner.sh` includes grep rules for known incident patterns (ctx.prisma, include: statements, wrong column names, etc.).

### Section 3: Failure Recurrence Analysis
Query memory for patterns with `recurrence_count > 1`. Any recurrence indicates a learning failure. Uses timeline analysis to detect patterns appearing more than once in a 30-day window.

### Section 4: Session Quality Statistics
Query session stats and prompt effectiveness metrics. Targets: average gaps per session < 5, recurrence rate = 0%, auto-learning compliance = 100%.

### Section 5: Gap Report and Remediation
Identify CRs without memory entries and incidents without scanner rules. For each gap:
- Missing memory: ingest via `massu_memory_ingest`
- Missing scanner rule: add grep check to `scripts/pattern-scanner.sh`

## Completion Criteria

- All CRs have >= 1 memory entry
- All grep-able incident patterns have scanner checks
- `recurrence_count == 1` for all patterns (no recurrences)
- Session stats reviewed, quality >= target
- All gaps remediated
- `pattern-scanner.sh` exits 0

## When to Use

- After incidents to verify the learning system captured them
- As a periodic check on auto-learning effectiveness
- When investigating whether a known bug pattern has been properly prevented
- After fixing bugs to confirm scanner rules were added
- When CR-34 compliance is in question

## Key Rules

- Proof over claims -- show MCP tool output, not summaries
- Every CR must have memory -- CRs without entries are learning gaps
- Every incident must have a scanner rule -- incidents without grep checks will recur
- Zero tolerance for recurrence -- any `recurrence_count > 1` is a learning failure
- FIX ALL gaps immediately (CR-9)
