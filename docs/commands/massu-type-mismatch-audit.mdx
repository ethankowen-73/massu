# /massu-type-mismatch-audit

Audit the entire codebase for type mismatches between frontend form schemas, backend router schemas, and database column types. Catches bugs that TypeScript cannot catch -- runtime Zod validation failures, database insert failures, and `as any` casts hiding structural mismatches.

## Usage

```
/massu-type-mismatch-audit                    # Full codebase audit (all 8 sections)
/massu-type-mismatch-audit [router-name]      # Audit a specific router and its consumers
```

## Audit Sections

### Section 1: Database Array Column Inventory
Identify all array-type columns and verify code sends arrays (not strings) to them. Query both Prisma schema and `information_schema` to find `text[]`, `int[]`, etc.

### Section 2: Frontend-Backend Zod Schema Comparison
Compare Zod schemas in frontend forms against backend router `.input()` schemas for the same procedure. Check field names, types, and optionality.

### Section 3: `as any` Cast Audit
Find all `as any` casts in API call contexts. Categorize: structural mismatch (P0), enum narrowing (P1), return type access (P1), or cosmetic (P2).

### Section 4: Number/Date/Boolean Coercion
Verify form fields are properly converted before sending to API. Check for `parseFloat`/`parseInt` on number inputs, ISO string conversion on dates, and boolean vs string mismatches.

### Section 5: Return Type / Serialization
Verify API responses can be serialized. Check BigInt values converted to `Number()`, Decimal columns converted, and dates returned as ISO strings.

### Section 6: Enum Value Alignment
Compare enum values in frontend against backend and database. Detect added/removed values that cause runtime validation failures.

### Section 7: Query Parameter Type Verification
Verify filter and query parameters match between frontend and backend. Check UUID validation, enum typing, and pagination parameter types.

### Section 8: Recurring Pattern Detection
Find systematic patterns: `(currentUser as any)` (missing return type), `useState<string>` for enum filters (should be union type), missing `.transform()` or `.preprocess()`.

## When to Use

- After any schema migration (new columns, type changes)
- After adding new API procedures or form components
- Before major releases
- When debugging "Expected X, received Y" Zod errors
- When debugging 400/500 errors on form submissions
- As part of periodic codebase audits

## Key Rules

- Trace the full path: frontend form to API call to Zod schema to DB column
- Always verify against `information_schema` for column types
- Every `as any` in an API call context is suspect
- Array columns MUST receive arrays, not strings
- Show proof: every finding includes file:line and the exact mismatch
- Uses parallel subagents for full codebase audits (4 subagents, 2 sections each)
