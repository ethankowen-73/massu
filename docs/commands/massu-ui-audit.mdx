# /massu-ui-audit

Execute a thorough UI/UX audit covering accessibility, responsiveness, component consistency, state management, and user flow completeness. Uses a mandatory loop controller that spawns fresh audit passes until zero issues are found.

## Usage

```
/massu-ui-audit
```

## Audit Sections (10 Sections)

### Section 1: Page Inventory
- List all pages, layouts, error boundaries, and loading states
- VR-RENDER: verify every component is rendered in at least one page
- Detect orphaned components (created but never used)

### Section 2: Navigation
- Analyze navigation components, extract all href values
- Verify Link components and router.push/replace calls

### Section 3: Buttons and Actions
- Inventory all buttons, verify handlers are attached
- Check for buttons without `type` attribute
- Flag dead buttons (no onClick, not type="submit", not disabled)

### Section 4: Forms
- Verify all forms have `onSubmit` handlers
- Check for react-hook-form and Zod validation usage

### Section 5: State Management
- Loading states: `isLoading`, `isPending`, Skeleton components, Suspense
- Empty states: proper messaging when data is absent
- Error states: error handling, toast notifications

### Section 6: Responsiveness
- Responsive class usage (`sm:`, `md:`, `lg:`, `xl:`)
- Mobile-specific code and media query hooks

### Section 7: Accessibility
- Images without alt text
- ARIA labels and describedby attributes
- Keyboard navigation handlers and tabIndex

### Section 8: Component Consistency
- Design system compliance (design tokens vs hardcoded colors)
- Button variant consistency
- Component reuse vs duplication

### Section 9: User Flows
- Identify and trace critical user flows end-to-end
- Verify each step: navigate, fill, submit, complete

### Section 10: UX Quality
- Focus ring visibility (VR-FOCUS)
- Touch target sizes >= 44x44px (VR-TOUCH)
- Motion preferences / prefers-reduced-motion (VR-MOTION)
- Error recovery flows and retry mechanisms (VR-RECOVERY)

## Mandatory Pattern Alignment

Before running checks, the audit verifies compliance with all UI patterns:
- DataTable for lists, sonner `toast()` (not useToast), dark mode contrast
- Select `__none__` placeholder (never `value=""`), responsive layout classes
- Loading/empty/error states present, form validation with Zod

## When to Use

- After significant UI changes or new page additions
- Before major releases for UX quality assurance
- When investigating accessibility or responsiveness issues
- As a periodic UI health check
- When onboarding new UI patterns to verify consistency

## Key Rules

- Every button must have a handler
- Every form must have validation
- Every async operation must have a loading state
- Every list must have an empty state
- Every action must have error handling
- FIX ALL issues encountered, including pre-existing ones (CR-9)
- The entire audit must pass in a SINGLE run with zero issues
