# /massu-production-verify

Confirm that deployed features actually work in production. "Deployed" is NOT "Working" -- this command verifies operational correctness by checking real data, real UI, and real API responses on the production environment.

## Usage

```
/massu-production-verify                           # Auto-detect from recent commits
/massu-production-verify "calendar sync"           # Verify specific feature
/massu-production-verify /path/to/plan.md          # Verify all items from a plan
/massu-production-verify --deferred                # Re-check pending deferred verifications
```

## Protocol Steps

### Phase 1: Determine What to Verify
- Identify recent changes from git log
- Categorize each change by verification type and timing (immediate vs deferred)
- Build verification matrix

### Phase 2: Immediate Verifications
- **Deployment health**: confirm latest deployment is READY via Vercel MCP
- **UI verification**: Playwright navigate, snapshot, console messages, interact
- **API/endpoint verification**: HTTP requests to production URLs
- **Database verification**: query NEW PROD for real data (not just table existence)
- **Environment and config**: verify feature flags and config values
- **Integration chains**: verify full trigger-process-store-display chain

### Phase 3: Deferred Verifications
For items that cannot be verified immediately (crons, webhooks, async jobs):
- Generate a deferred checklist with verification queries and deadlines
- Save to `session-state/deferred-verifications.md`
- Deferred items are surfaced by `/massu-bearings` in the next session

### Phase 4: Results and Report
- Produce verification report with proof for each check
- Status: VERIFIED, VERIFIED + DEFERRED, PARTIALLY VERIFIED, or FAILED

### Phase 5: Follow-Up
- If any immediate check fails: diagnose, fix, push, and re-verify
- Deferred items checked via `/massu-production-verify --deferred`

## When to Use

- After every deployment to production
- When a feature is marked "complete" but needs production validation
- To check deferred items from a previous verification
- When users report something is not working after a deploy
- Before declaring a plan fully complete

## Key Rules

- Never verify against DEV -- production verification means NEW PROD and production URLs
- Don't confuse schema with data -- a table existing is meaningless if no data flows through it
- Never write or modify production data during verification -- query only
- All timestamps in PST (user timezone)
- Read-only on production -- no mutations
